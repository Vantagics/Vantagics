# 报告生成系统多语言本地化分析

## 执行摘要

报告生成系统（包括PDF、Word、Excel、PPT导出）存在严重的多语言本地化问题。系统中存在大量硬编码的中文字符串，包括：
- 文件对话框标题
- 报告章节标题
- LLM提示词
- 错误消息
- 页面元素文本

这些硬编码字符串导致英文用户无法正常使用报告生成功能。

## 问题详情

### 1. 文件对话框标题（高优先级）

**位置**: `src/app_report_generate.go`

```go
// 当前代码（硬编码中文）
runtime.SaveFileDialog(a.ctx, runtime.SaveDialogOptions{
    Title: "保存分析报告",  // ❌ 硬编码
    DefaultFilename: fmt.Sprintf("分析报告_%s.docx", timestamp),  // ❌ 硬编码
})
```

**影响**: 英文用户看到中文对话框标题和文件名

**建议修复**:
```go
// 使用国际化
runtime.SaveFileDialog(a.ctx, runtime.SaveDialogOptions{
    Title: i18n.T("report.save_dialog_title"),
    DefaultFilename: fmt.Sprintf("%s_%s.docx", 
        i18n.T("report.filename_prefix"), timestamp),
})
```

### 2. LLM提示词（高优先级）

**位置**: `src/app_report_generate.go:531-560`


**问题**: 整个系统提示词完全用中文编写，长达30多行

```go
systemPrompt := `你是一位专业的数据分析报告排版专家。你的唯一任务是将下方提供的分析结果直接整理成一份格式规范的正式报告。

【最重要的规则】
- 所有需要的数据已经完整提供在下方，不需要获取任何额外信息
- 必须直接输出报告正文，第一行就是 # 标题
...
`
```

**影响**: 
- 英文用户生成的报告仍然是中文结构
- LLM无法理解英文用户的需求
- 报告章节标题固定为中文

**建议修复**:
1. 将提示词模板化并国际化
2. 根据用户语言设置选择对应的提示词
3. 报告章节标题应该根据语言动态生成

## 风险和缓解措施

### 风险1: LLM提示词翻译质量
**描述**: 英文提示词可能无法达到中文提示词的效果
**缓解**: 
- 请母语者审查英文提示词
- 进行充分的LLM输出测试
- 必要时调整提示词措辞

### 风险2: 报告格式不一致
**描述**: 不同语言生成的报告格式可能不一致
**缓解**:
- 使用相同的报告结构模板
- 测试两种语言的输出
- 确保章节标题对应

### 风险3: 字体支持问题
**描述**: PDF中英文字体可能显示不正确
**缓解**:
- 使用支持多语言的字体（如Arial Unicode MS）
- 测试不同字体的显示效果
- 提供字体回退机制

### 风险4: 日期时间格式
**描述**: 不同语言的日期时间格式不同
**缓解**:
- 根据语言设置选择格式
- 使用标准的日期时间格式化
- 测试边界情况
